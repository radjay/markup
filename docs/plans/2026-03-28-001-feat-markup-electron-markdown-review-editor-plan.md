# Markup — Native Markdown Review Editor for AI Agent Feedback

## Overview

Markup is an Electron-based native desktop app for reviewing markdown documents generated by AI coding agents (Claude Code, Cursor, etc.). It provides a file sidebar for navigation, a beautifully rendered reading pane, inline and document-level commenting, direct editing, and saves feedback back into the markdown file so the coding agent can parse it and continue.

## Problem Statement / Motivation

When AI coding agents generate plans, requirements, or design documents as markdown files, the review workflow is broken:

- **Cursor/VS Code** can pretty-render markdown but have no inline commenting that saves back to the file
- **Obsidian** is a knowledge management tool with plugin-based commenting — overkill and non-standard formats
- **Typora/iA Writer** are beautiful editors but have zero annotation capabilities
- **HackMD/Notion/Outline** store comments in their platform layer — comments are lost on export
- **md-review** (closest competitor) is browser-based, stores comments separately, requires manual copy-paste back

No existing tool completes the full loop: **agent generates markdown → human reviews with comments → comments saved in-file → agent reads feedback and continues**. This is the gap Markup fills.

## Competitive Landscape

| Tool | Rendering | Inline Comments | Comments in File | Native App | Agent Loop |
|------|-----------|----------------|-----------------|------------|------------|
| Cursor + plugins | Good | No | N/A | Yes | No |
| Obsidian + Commentator | Good | CriticMarkup | Yes | Yes (Electron) | No |
| Typora | Excellent | No | N/A | Yes (Electron) | No |
| md-review | Good | Yes | Separate store | No (browser) | Manual copy |
| HackMD | Good | Yes | Lost on export | No (web) | No |
| Markdown Docs (VS Code ext) | Good | Yes | Unclear format | No (extension) | Partial |
| **Markup** | **Excellent** | **Yes** | **Yes** | **Yes (Electron)** | **Yes** |

### Key Differentiators
1. Purpose-built for the AI agent feedback loop — not a general editor
2. Comments serialized directly in the markdown file in a parseable format
3. Native desktop app with file sidebar (not browser-based)
4. Beautiful rendered reading view — review experience, not a code editor
5. Claude Code plugin/skill that teaches agents how to parse Markup comments

## Comment Serialization Format

This is the core design decision. Comments must be:
- **Machine-parseable** by coding agents
- **Invisible** in standard markdown renderers
- **Resilient** to partial file rewrites by agents
- **Self-describing** so orphaned comments are still readable

### Format: HTML Comments with Structured Metadata

**Inline comments** — inserted immediately after the anchored content:

```markdown
## Architecture

The system uses a microservices architecture with event-driven communication.

### Database Layer

We'll use PostgreSQL with read replicas.

```

**Document-level comments** — placed at the end of the file in a dedicated block:

```markdown

```

**Review metadata** — in YAML frontmatter (appended to existing frontmatter if present):

```yaml
---
# ... existing frontmatter preserved ...
markup_reviewed: true
markup_reviewer: radjay
markup_reviewed_at: 2026-03-28T10:36:00Z
markup_status: changes_requested  # approved | changes_requested | commented
---
```

### Anchoring Strategy

Inline comments anchor to the **nearest preceding heading** (e.g., `## Architecture`, `### Database Layer`). This is:
- **Resilient** — headings are the most stable part of a markdown document
- **Human-readable** — even if parsed raw, you can see what section the comment refers to
- **Coarse-grained by design** — section-level feedback is the natural unit for plan review

For finer-grained anchoring (specific text selection), the comment includes an optional `selection` field with a text snippet:

```html

```

### What Happens When Content Changes

- If the agent rewrites a section, comments anchored to that heading survive (heading still exists)
- If the agent removes a heading, comments become orphaned but remain readable in the file
- On reload, Markup displays orphaned comments in a "Resolved/Orphaned" section
- The agent can explicitly clear comments by removing the HTML comment blocks

## User Flows

### Flow 1: Opening the App

**Entry points:**
1. Launch app → empty state → "Open File" or "Open Folder" prompt
2. CLI: `markup plan.md` or `markup ./docs/plans/`
3. Drag-and-drop file/folder onto app icon or window
4. OS "Open With" file association for `.md` files
5. Recent files list on welcome screen

### Flow 2: Sidebar Navigation

- Shows directory tree filtered to `.md` files
- Collapsible folders
- Current file highlighted
- Below the file tree: heading-based outline/TOC for the current document
- Click a heading → scrolls main pane to that section

### Flow 3: Reading Rendered Markdown

- Default mode is **Review Mode** — beautiful rendered markdown
- Supports: headings, lists, code blocks (syntax highlighted), tables, task lists, blockquotes, links, images
- Stretch: mermaid diagrams, math (KaTeX)
- Existing `@markup` comments are rendered as colored markers/badges in the margin
- Hovering a marker shows the comment text

### Flow 4: Adding an Inline Comment

1. User selects text or clicks a section heading
2. A comment popover appears (or sidebar panel activates)
3. User types comment and submits
4. Comment appears as a colored marker in the margin
5. Comment is stored in-memory, file marked as "unsaved"

### Flow 5: Adding a Document-Level Comment

1. User clicks "Add Comment" button in the document comments panel
2. Types general feedback
3. Comment appears in the document comments list
4. File marked as "unsaved"

### Flow 6: Editing Content Directly

1. User toggles from Review Mode → Edit Mode
2. Main pane switches to CodeMirror 6 with raw markdown + syntax highlighting
3. User makes changes
4. Toggling back to Review Mode shows the updated rendered content
5. File marked as "unsaved"

### Flow 7: Saving

1. User presses Cmd+S or clicks Save
2. Markup serializes:
   - Direct edits → applied to markdown content
   - Inline comments → HTML comments inserted after their anchor sections
   - Document comments → HTML comment block at end of file
   - Review metadata → appended to frontmatter
3. File written atomically (write to `.tmp`, then rename)
4. "Unsaved" indicator clears

### Flow 8: File Watching (External Changes)

- Markup watches the open file with `fs.watch`
- If the file changes externally (agent rewrites it):
  - Notification: "This file was modified externally. Reload?"
  - Reload: re-parse, preserve any unsaved comments if possible
  - Ignore: keep current state (warn that saving will overwrite)

### Flow 9: Unsaved Changes Guard

- Closing window or switching files with unsaved changes → "Save / Discard / Cancel" dialog
- Auto-save option in preferences (saves on every comment/edit after a debounce)

## Claude Code Plugin / Skill

A critical part of the Markup ecosystem: a Claude Code plugin or skill that teaches coding agents how to read and respond to Markup comments.

### What It Does

1. **Parses `@markup` comments** from markdown files the agent has written
2. **Presents structured feedback** to the agent: which section, what type (inline/document), the comment text
3. **Guides the agent's response** — update the plan, address each comment, remove resolved comments

### Plugin Structure

```
markup-claude-code-plugin/
  CLAUDE.md              # Instructions for Claude Code
  skill.md               # Skill definition
  lib/
    parse-markup.ts      # Parser for @markup HTML comments
```

### Skill Behavior

When invoked (e.g., `/markup-review plan.md` or triggered automatically when a file contains `@markup` comments):

1. Read the markdown file
2. Extract all `@markup` inline comments and `@markup-doc-comments` blocks
3. Present to the agent as structured feedback:
   ```
   ## Review Feedback for plan.md
   Status: changes_requested
   Reviewer: radjay

   ### Inline Comments
   1. [## Architecture] "This is over-engineered for our scale. Consider a modular monolith instead."
   2. [### Database Layer] "Good choice. Make sure to specify connection pooling strategy."

   ### Document Comments
   1. "Overall the plan is solid but the timeline in Phase 2 is too aggressive."
   2. "Missing error handling strategy — please add a section."
   ```
4. Instruct the agent to address each comment, update the relevant sections, and remove resolved `@markup` comment blocks
5. After the agent updates the file, Markup detects the change via file watching and reloads

### CLAUDE.md Instructions (for the plugin)

```markdown
# Markup Comments

When you encounter HTML comments in markdown files matching the pattern
``, these are review comments left by a human
using the Markup editor.

## How to handle them:
1. Parse each comment's JSON metadata to understand which section it targets
2. Read the comment text (everything after the closing `}`)
3. Address the feedback by updating the relevant section
4. Remove the `@markup` comment block once addressed
5. Do NOT remove `@markup-doc-comments` blocks — the reviewer will clear these
6. Preserve any frontmatter fields prefixed with `markup_`
```

## Technical Considerations

### Security
- `contextIsolation: true`, `sandbox: true`, `nodeIntegration: false` (Electron defaults)
- All file I/O through validated IPC handlers in main process
- Path traversal prevention on all file operations
- Content Security Policy for renderer

### Performance
- Tiptap/ProseMirror handles large documents well (virtual rendering)
- CodeMirror 6 uses viewport-based rendering — no performance cliff on large files
- File tree lazy-loads subdirectories
- Atomic file writes (`.tmp` + rename) prevent data loss

### Comment Format Resilience
- HTML comments survive any markdown renderer/processor
- Heading-based anchoring survives most content edits
- Self-describing format — even orphaned comments are human-readable
- Agent can clear resolved comments by removing the HTML comment blocks

## System-Wide Impact

### Interaction Graph
- User opens file → main process reads file → preload bridges to renderer → remark parses markdown + extracts `@markup` comments → Tiptap renders with comment decorations
- User saves → renderer serializes comments → preload bridges to main → main writes atomically → `fs.watch` on agent's side detects change
- Agent updates file → `fs.watch` in main process fires → renderer prompted to reload

### Error Propagation
- File read failures surface as user-visible error in the main pane
- File write failures show native dialog with retry option
- Malformed `@markup` comments are preserved as-is (don't crash, don't discard)
- IPC errors caught at the preload boundary

### State Lifecycle Risks
- **Partial save**: Atomic write (tmp + rename) prevents corrupted files
- **Concurrent edits**: File watch + reload prompt prevents silent overwrites
- **Comment orphaning**: Orphaned comments preserved and shown in UI, not silently dropped

## Success Metrics

- A user can review a Claude Code-generated plan, leave comments, save, and Claude Code can parse and address the feedback — all within 2 minutes
- Comments survive round-trips: user comments → agent rewrites → user re-reviews
- File remains valid markdown at all times (viewable in any renderer)

## Dependencies & Risks

| Risk | Mitigation |
|------|------------|
| Tiptap comment extensions may be Pro-only | Build custom ProseMirror marks for comment anchoring |
| Comment format may need iteration | Keep format simple, version it (`@markup` prefix), maintain backwards compat |
| Agent may strip HTML comments when rewriting | Document in Claude Code plugin that `@markup` comments should be preserved until addressed |
| Electron app size (~150MB+) | Accept for v1; explore Tauri for v2 if size is a concern |
